  <action>
Create the shared multiplayer types first (src/shared/types/multiplayer.ts):
- `GameStateUpdate`: board matrix (number[][]), current piece, score, level, lines cleared, alive status
- `GarbageEvent`: fromPlayerId, toPlayerId, lineCount, gapColumn, timestamp
- `PlayerState`: playerId, nickname, gameState (GameStateUpdate), connected, eliminated
- `MatchState`: roomId, players (Map or Record of PlayerState), status (countdown|playing|finished), startedAt

Then build the sync manager (src/server/multiplayer/syncManager.ts):
- Class `SyncManager` that tracks active matches by roomId
- `startMatch(roomId, playerIds)`: initializes MatchState for all players
- `updatePlayerState(roomId, playerId, state: GameStateUpdate)`: stores latest state and broadcasts to all other players in the room via WebSocket
- `getMatchState(roomId)`: returns current MatchState
- Broadcast uses the existing WebSocket infrastructure from Phase 1 room system
- State updates are delta-friendly: only send changed fields if possible, but full state is acceptable for v1

Wire into WebSocket handlers (src/server/handlers/gameHandlers.ts):
- `game:state` event: player sends their board state, sync manager broadcasts to others
- `game:start` event: triggers match initialization via sync manager
- `game:lineClear` event: triggers garbage calculation (wired in Task 2)
- Keep handlers thin — delegate logic to syncManager and garbageEngine

Target: ~60fps state updates (every ~16ms) or throttled to every 50ms to reduce bandwidth. Use throttling per player.

  <behavior>
    - calculateGarbage("single") returns 0 lines
    - calculateGarbage("double") returns 1 line
    - calculateGarbage("triple") returns 2 lines
    - calculateGarbage("tetris") returns 4 lines
    - calculateGarbage("tspin_single") returns 2 lines
    - calculateGarbage("tspin_double") returns 4 lines
    - calculateGarbage("tspin_triple") returns 6 lines
    - resolveTarget(senderId, players[]) returns a valid opponent playerId (not sender, not eliminated)
    - GarbageQueue.add(lines) queues garbage, GarbageQueue.cancel(lines) reduces pending count
    - GarbageQueue after delay (~1.5s) emits garbage for board insertion
    - Full cancel: if player clears lines while garbage pending, pending is reduced first before sending garbage to others
  </behavior>
  <action>
Build the server-side garbage engine (src/server/multiplayer/garbageEngine.ts):
- `calculateGarbage(clearType: ClearType): number` — standard guideline table per user decision: Single=0, Double=1, Triple=2, Tetris=4, T-spin variants scale (tspin_single=2, tspin_double=4, tspin_triple=6)
- `resolveTarget(senderId: string, players: PlayerState[]): string` — auto-target: pick random alive opponent or last attacker. Per user decision, no manual targeting (v2)
- `createGarbageEvent(senderId, targetId, lineCount): GarbageEvent` — generates event with random gap column (0-9)
- Combo multiplier: consecutive clears increase garbage by +1 per combo step
- Back-to-back bonus: +1 line for consecutive difficult clears (tetris or T-spin)

Build the client-side garbage queue (src/engine/garbageQueue.ts):
- Class `GarbageQueue` with internal pending array of {lines, gapColumn, receivedAt}
- `add(event: GarbageEvent)`: adds to pending queue
- `cancel(lines: number)`: reduces pending garbage (full cancel per user decision). Returns remaining cancel amount if more lines cleared than garbage pending
- `tick(now: number)`: checks if any pending garbage has exceeded delay (~1500ms), returns garbage entries ready to insert into board
- `insertGarbage(board, garbageEntry)`: pushes rows from bottom of board — each row is filled except for the gap column. Board shifts up, top rows that overflow are lost
- Garbage queue visible indicator: expose `getPendingCount()` and `getPendingEntries()` for UI rendering

Wire garbage into game handlers: when `game:lineClear` received, call calculateGarbage, resolveTarget, create event, send to target player via WebSocket.
